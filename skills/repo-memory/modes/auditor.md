# Auditor Mode

Use this mode when checking Repo Memory integrity, identifying documentation drift, and inspecting schema compliance.

## Instructions

1. **Config audit:** Verify that the configuration file (`repo-memory.config.yml`) exists, is syntactically valid, matches version 3, and resolves all paths correctly.
2. **Ownership audit:** Ensure the canonical ownership map exists (typically `docs/README.md`) and every capability is explicitly mapped.
3. **Directory structure audit:** Verify that all expected directories (features, generated, intake, etc.) exist or have their paths mapped.
4. **Frontmatter validation:** Verify YAML frontmatter fields, types, and values in:
   - Feature docs (`id`, `title`, `status`, `priority`, `ready`, `owner`, `canonical_doc`, `next_safe_step`, `blocked_by`, `evidence`, `confidence`).
5. **No duplicate keys:** Ensure feature IDs and canonical docs are globally unique across all directories.
6. **Central files check:** Check for legacy files (e.g. `docs/feature-registry.md`) and verify they are marked deprecated or generated.
7. **Generated files header:** Verify that generated files under the generated path start with `<!-- Generated by repo-memory. Do not edit directly. -->`.
8. **Adapter verification:** Verify that active agent files (e.g. `AGENTS.md`, `CLAUDE.md`) contain managed block delimiters and are thin routing files.
9. **Evidence audit:** Scan for inferred claims and ensure they list supporting evidence files.
10. **Link validation:** Check for broken local markdown links and anchors within docs.
11. **Run validation:** Prefer running `python3 <skill-dir>/scripts/validate-docs.py --project-docs .` to execute these checks programmatically.
